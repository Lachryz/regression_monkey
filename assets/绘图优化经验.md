# 绘图优化经验

本文记录 `regression_monkey` 在 PNG 与交互式 HTML 绘图中的设计和性能经验，供后续修改 `src/regression_monkey/plot/png.py`、`src/regression_monkey/plot/html.py` 以及相关导出逻辑时参考。

## 核心目标

规格曲线图需要同时满足三类需求：

- 小规模图表要方便解释：能看清单个规格的系数、显著性、控制变量和样本量。
- 大规模图表要保留整体形态：即使规格数达到数千到上万，也不能因为列宽无限压缩而变成不可读的噪声。
- PNG 与 HTML 要共享视觉语言：显著性颜色、特殊规格标记、参考线、控制变量矩阵和标题元数据应尽量一致。

因此 HTML 不应只做成“可交互 PNG”，而要拆分为两种视图：`DETAIL` 负责解释单个规格，`COMPACT` 负责承载大规模总览。

## DETAIL 与 COMPACT 的分工

`DETAIL` 是完整交互模式：

- 保留右侧详情栏，用于展示当前规格纳入的控制变量、控制变量自身统计量、样本量、R2、F 统计量等信息。
- 支持悬停高亮、点击固定、再次点击或 `Esc` 取消固定。
- 当规格列数超过可视宽度时，只让面板内容横向滚动，面板边框、左侧标签和右侧面板标签保持稳定。
- 不显著规格在 STAR 面板中用带方向的空心单元格表示，正系数在中心线上方，负系数在中心线下方。

`COMPACT` 是大规模总览模式：

- 默认只在规格数大于 1024 时启用并进入；较小图表直接进入 `DETAIL`，避免用户看到一个被过度压缩但交互信息较少的视图。
- 隐藏右侧详情栏，关闭中央图表悬停/点击选择和紫色选中列覆盖层。
- 列宽缩小到 8192 个规格对应的基准宽度后不再继续压缩；规格更多时改用横向滚动。
- STARS、CONTROL、OBS 使用接近 PNG 的连续线条或细方块，减少大量独立圆角矩形带来的绘制成本和视觉碎片。
- COEF 点在两种模式下都应更小；在 `COMPACT` 中继续按基准列宽缩放。

这个分工的关键不是简单地“开关侧栏”，而是明确两种阅读任务：`DETAIL` 读单列，`COMPACT` 读全局形态。

## 布局与滚动

交互式 HTML 的布局最容易出问题的地方是“固定信息”和“可滚动内容”混在一起。建议遵守以下规则：

- 先分配右侧详情栏，再动态计算画布主绘图区宽度。
- 左侧刻度、控制变量标签和 alternative-group 标记应覆盖在固定白色背景上，避免横向滚动内容压到标签上。
- `DETAIL` 中如果规格列超过当前宽度，只滚动各面板内部内容；面板框线、标题、坐标标签和右侧旋转面板名不要跟着跑。
- `COMPACT` 中允许整体图形横向滚动，但列宽必须有下限，不能无限缩小。
- 排序、筛选、Guide/CI 开关、窗口尺寸变化都会改变可见内容或比例，应触发重绘；单纯横向滚动不应全量重绘。

## 性能缓存

大规模 HTML 图的核心性能策略是把 `COMPACT` 视图缓存为一张位图：

- 首次渲染 `COMPACT` 后，把完整紧凑图缓存为 bitmap。
- 横向滚动时只裁切和绘制可见切片。
- 排序、筛选 chips、Guide chips、CI chips、视口尺寸变化或模式切换会使缓存失效，需要重建。
- 不要在滚动事件中重新计算所有规格的几何元素，也不要为每个规格保留过多独立 DOM 节点。

`DETAIL` 仍然可以保留逐规格交互，因为它服务的是阅读和诊断；`COMPACT` 则应把交互开销降到最低。

## 用 Canvas 替代大量 SVG / DOM

当规格数很大时，性能瓶颈通常不在单个几何元素的计算，而在浏览器维护大量 SVG 节点或 DOM 节点：

- 每个规格的 STAR 单元格、COEF 点、CI 带、CONTROL 单元格、OBS bar 如果都生成独立 SVG 元素，几千到上万个规格会迅速变成数万甚至数十万个节点。
- 排序、筛选、hover、resize 时，浏览器需要重新计算这些节点的样式、布局、命中区域和绘制顺序。
- 即使单个元素很简单，大量节点也会导致滚动卡顿、交互延迟和内存占用上升。

因此大规模图表更适合用 Canvas 绘制主图层：

- Canvas 把图表看作像素绘制任务，不为每个规格保留一个可交互节点。
- 连续的 STARS、CONTROL、OBS 标记可以批量绘制为线段、矩形或路径。
- COEF 点和 CI 切片可以按当前排序后的数组顺序一次性绘制。
- 横向滚动时可以只重绘可见范围，或在 `COMPACT` 中直接裁切已缓存 bitmap。

推荐的混合架构是：

- Header、按钮、selector、chips、右侧详情栏继续使用 HTML DOM，便于无障碍状态、键盘操作和文本排版。
- 主图表区域使用 Canvas 绘制，尤其是规格列密集的 STARS、COEF、CONTROL、OBS 面板。
- 需要固定清晰文字的区域，如左侧控制变量标签、坐标刻度、右侧旋转面板标签，可以用 DOM / SVG / Canvas 任一方式，但必须和滚动内容分层，避免被覆盖。
- 交互命中不要依赖 DOM 节点，而应由鼠标坐标反算规格索引，再用当前排序数组查记录。

Canvas 替代方案的关键收益是把复杂度从“节点数量”转为“像素绘制次数”。这对 `COMPACT` 尤其重要，因为 `COMPACT` 本身不需要逐列 hover/click，完全可以把中央图表当作不可交互的高密度图像来渲染。

但 Canvas 也有边界：

- 文字渲染和自动换行能力弱于 DOM，长标签、详情文本和工具栏不应强行搬进 Canvas。
- Canvas 本身没有元素级可访问性和浏览器原生命中状态，交互逻辑需要手写。
- 高 DPI 屏幕需要按 `devicePixelRatio` 放大 backing store，否则线条和文字会发虚。
- 每次 resize 都要重新计算几何布局和重绘，不要假设 CSS 缩放可以替代真实重绘。
- 如果图表要导出为静态图片，需要确认 Canvas 图层、DOM 标签和叠加层都能被一致地截图或重新绘制。

实践上，可以把 `DETAIL` 保持为“Canvas 主图 + DOM 详情”的交互图，把 `COMPACT` 进一步优化为“Canvas 绘制一次 + bitmap 缓存 + 滚动裁切”。这样既能减少节点压力，又不会牺牲小规模图表的解释能力。

## 视觉编码

绘图优化时要优先保护已经形成的视觉约定：

- 显著性图例颜色必须与 COEF 点颜色一致，`n.s.` 使用黑色。
- STARS 中正向显著与负向显著使用方向颜色，选中状态统一覆盖为紫色。
- `Full controls`、`No controls_test` 等 helper lines 要有稳定颜色，并且只在相应 Guide chip 关闭时隐藏。
- COEF 置信区间用每个规格的切片绘制，不要合并成一整个平滑多边形，否则边缘会过度平滑，和 PNG 观感不一致。
- STARS 中心线、COEF 零线、OBS 均值线使用一致的红色虚线参考线。
- CONTROL 面板中的 alternative-group I 形标记应放在预留左侧 gutter 中，避免覆盖变量名；可选组虚线标记要和变量标签同色，邻近组之间要有可见断点。
- 如果所有 3 星显著规格都包含某个 `controls_test` 变量，该控制变量行标签可用黄色背景提示，但不要影响单元格本身的显著性或选中状态编码。

这些规则的目的，是让用户在 PNG、DETAIL、COMPACT 之间切换时，不需要重新学习图例。

## 交互状态

HTML 顶部工具栏和图表内部选中态必须分层管理：

- SORT 按钮、Mode 按钮、Guide chips、CI chips 属于 header state。
- 规格 hover、pin、紫色选中列属于 chart state。
- 清除图表选中态时，只能移除 `#chart` 内部元素的 `.active`，不要误清 header 控件。
- 清除 chart state 后，需要从当前排序状态重新同步 SORT 按钮的 `.active` 和 `aria-pressed="true"`。
- `COMPACT` 是非交互图表展示模式，不应保留 `DETAIL` 的 hover/click 选择行为。

这条规则能避免一个常见回归：用户点击空白处清除规格选中后，顶部排序按钮也失去白色 active 状态。

## 数据与标题

绘图层应尽量消费标准 handoff 文件，而不是重新依赖原始数据：

- `*_results.csv` 保存每个规格的估计结果、显著性、样本量、控制变量集合和统计量。
- `*_plot_meta.json` 保存标题、控制变量结构、规格名、引擎、耗时、排序所需元数据等。
- HTML 排序应在页面内处理，不要为了不同排序模式预先写出不同记录顺序。
- HTML 切换排序时，应复用初始 SVG / payload 中保存的系数轴范围，例如 `data-coef-min` 和 `data-coef-max`，不要根据已绘制点坐标反推比例。
- PNG 和 HTML 标题元数据都应显示分析引擎；多 Y、多 X、多规格时，HTML 使用顶部 selector bar 切换。
- 规格标题面向用户展示 Stata 风格 `absorb(...) vce(...)`，内部 flag、tag、文件夹名和配置键保持稳定不变。

绘图层和分析层的边界越清楚，后续重画、调试、HTML 单独导出和跨引擎一致性就越容易维护。

## 验证清单

修改绘图逻辑后，建议至少检查以下场景：

- 小规格数图表：应默认进入 `DETAIL`，`COMPACT` 禁用或不可用。
- 大规格数图表：应默认进入 `COMPACT`，横向滚动顺滑，滚动不触发全量重绘。
- 多 Y / 多 X / 多 spec：顶部 selector 能切换，标题和详情栏随之更新。
- 三种排序：`coef`、`obs`、`sig(coef)/p` 都能工作，默认排序按钮保持 active。
- Guide chips：`ALL TEST` 和 `NO TEST` 只隐藏或显示对应 helper lines。
- CI chips：置信区间显示切换不改变系数轴含义。
- DETAIL hover/click：STAR、COEF、CONTROL、OBS 面板同步高亮；点击固定和 `Esc` 清除正常。
- COMPACT 模式：无右侧详情栏，无中央 hover/click 选中态，无紫色选中列覆盖层。
- CONTROL 面板：控制变量顺序保持用户输入顺序，alternative-group 标记不遮挡文字。
- 截图检查：桌面宽屏和较窄窗口下不应出现标题、标签、面板框线和滚动内容互相覆盖。

如果只改了视觉样式，也应至少打开一次生成的 HTML 手工检查 `DETAIL` 与 `COMPACT`，因为很多问题只有在真实浏览器布局和 canvas 缩放下才会出现。
